## 1. Frontend Development

### Frontend Development란
- 웹사이트/웹 애플리케이션의 **UI와 UX를 만들고 디자인하는 것**
- HTML, CSS, JavaScript를 활용해 사용자가 **직접 상호작용하는 부분**을 개발

### Client-side frameworks
- **클라이언트 측에서 UI와 상호작용을 개발하기 위해 사용되는 JavaScript 기반 프레임워크**
- 필요한 이유
  1. **웹에서 하는 일이 많아졌다** — 문서 읽기 → 스트리밍, 영상, 실시간 채팅 등 복잡한 대화형 **웹 애플리케이션(web applications)** 시대
  2. **다루는 데이터가 많아졌다** — 동일한 데이터가 화면 여러 곳에 출력될 때, 데이터와 화면을 일치시키는 작업이 필요
- Vanilla JS의 한계: 데이터가 바뀔 때마다 `querySelector`로 요소를 찾아 `textContent`를 일일이 갱신해야 함 → 데이터 추적·업데이트(렌더링/추가/삭제)를 자동화하는 도구 필요

### SPA (Single Page Application)
- **단일 페이지로 구성된 애플리케이션**
- 동작마다 페이지 전체를 새로 로드하지 않고, 사용자가 상호작용 할 때마다 JS 프레임워크가 클라이언트 측에서 UI 렌더링을 관리 → **CSR 방식 사용**

### CSR (Client-side Rendering)
- 클라이언트(브라우저)에서 화면을 렌더링하는 방식
- 동작 과정
  1. 브라우저가 서버로부터 **최소한의 HTML + JS 파일** 로드
  2. JS가 실행되며 필요한 **데이터를 서버에 요청**
  3. 서버는 완성된 HTML이 아닌 **데이터(JSON)만 응답**
  4. 클라이언트의 JS가 데이터를 바탕으로 **DOM을 렌더링**

| 장점 | 단점 |
|---|---|
| **빠른 페이지 전환** — 데이터만 받아 일부만 재렌더링, 서버 부하 감소 | **느린 초기 로드** — JS 다운로드·파싱·실행 전까지 완전한 렌더링 불가 |
| **뛰어난 사용자 경험** — 새로고침 없음, 네이티브 앱과 유사 | **SEO 문제** — 콘텐츠를 나중에 그리므로 검색 엔진이 HTML 분석 시 내용이 비어 있음 -> NEXT(?)등의 해결방법이 있음|
| **FE/BE의 명확한 분리** — FE는 UI·상호작용, BE는 데이터·API → 대규모 앱 개발/유지관리 용이 | |

### SPA vs MPA / CSR vs SSR
- **MPA (Multi Page Application)**: 여러 개의 HTML 파일을 서버로부터 각각 로드. 페이지 이동 시마다 새 HTML 로드
- **SSR (Server-side Rendering)**: 서버에서 화면을 렌더링. 모든 데이터가 담긴 완성된 HTML을 서버가 만들어 클라이언트에 전달

## 2. Vue

### What is Vue
- **사용자 인터페이스를 구축하기 위한 JavaScript 프레임워크**
- Evan You가 개발 (2014 출시, Angular 개발팀 출신, 미술/디자인 배경)
- 최신 버전은 **Vue 3** — 공식 문서 https://vuejs.org (Vue 2 문서에 접속하지 않도록 주의)

### Vue의 2가지 핵심 특징
1. **선언적 렌더링 (Declarative Rendering)** — 표준 HTML을 확장한 **템플릿 구문**으로, JS 상태(데이터)를 기반으로 화면에 출력될 HTML을 선언적으로 작성
2. **반응성 (Reactivity)** — JS 상태(데이터) 변경을 추적하고, 변경이 발생하면 **자동으로 DOM을 업데이트**

### Vue 체험하기 (CDN)
```html
<div id="app">
  <h1>{{ message }}</h1>
</div>

<script src="https://unpkg.com/vue@3/dist/vue.global.js"></script>
<script>
  const { createApp, ref } = Vue

  const app = createApp({
    setup() {
      const message = ref('Hello vue!')
      return { message }
    }
  })
  app.mount('#app')
</script>
```

### Vue Style Guide
Vue 스타일 가이드 규칙은 우선순위에 따라 **4가지 범주**로 구분

| 범주 | 성격 |
|---|---|
| A. 필수 (Essential) | 오류를 방지하므로 어떤 경우에도 준수 |
| B. 적극 권장 (Strongly Recommended) | 가독성·개발자 경험 향상 |
| C. 권장 (Recommended) | 일관성 유지를 위해 권장 |
| D. 주의 필요 (Use with Caution) | 잠재적 위험이 있는 기능, 신중히 사용 |

> 참고: https://vuejs.org/style-guide

## 3. Vue Tutorial (CDN 방식으로 시작하기)

### 첫 Vue 작성하기 — 6단계
1. **CDN 작성** — `<script src="https://unpkg.com/vue@3/dist/vue.global.js">`
2. **createApp 가져오기** — `const { createApp } = Vue`
3. **Application instance** — 모든 Vue 앱은 `createApp` 함수로 새 인스턴스를 생성하는 것으로 시작
4. **Root Component** — `createApp`에는 객체(컴포넌트)가 전달됨. 모든 앱에는 다른 컴포넌트를 포함할 수 있는 **최상위(root) 컴포넌트**가 필요
5. **Mounting the App** — `app.mount('#app')`으로 HTML 요소에 앱 인스턴스를 탑재(연결). **인스턴스당 `mount()`는 한 번만 호출 가능**
6. 전체 구조 완성

```html
<div id="app"></div>

<script src="https://unpkg.com/vue@3/dist/vue.global.js"></script>
<script>
  const { createApp } = Vue
  const app = createApp({
    setup() { ... }
  })
  app.mount('#app')
</script>
```

### 반응형 상태 — `ref()`
- **반응형 상태(데이터)를 선언하는 함수** (ref === **reactive reference**, 반응형을 가지는 참조 변수를 만드는 것)
- 인자를 받아 **`.value` 속성이 있는 ref 객체로 래핑(wrapping)하여 반환**
- ref로 선언된 변수의 값이 변경되면, 해당 값을 사용하는 **템플릿에서 자동으로 업데이트**
- 인자는 **어떠한 타입도 가능**

```js
const { createApp, ref } = Vue

const app = createApp({
  setup() {
    const message = ref('Hello vue!')
    console.log(message)        // ref 객체
    console.log(message.value)  // 'Hello vue!'
  }
})
```

- **템플릿에서는 `.value`를 작성할 필요 없음** — 자동으로 벗겨짐 (automatically unwrapped)

### Vue 기본 구조
- `createApp()`에 전달되는 객체는 **Vue 컴포넌트(Component)**
- 컴포넌트의 상태는 **`setup()` 함수 안에서 선언**하고, 템플릿에서 쓰려면 **객체로 반환(return)** 해야 함

```js
const app = createApp({
  setup() {
    const message = ref('Hello vue!')
    return { message }
  }
})
```

### 템플릿 렌더링
- **Mustache 구문(콧수염 구문) `{{ }}`** 으로 상태 값을 기반으로 동적 텍스트 렌더링
- 템플릿 안에서는 **JavaScript 표현식도 사용 가능**

```html
<h1>{{ message }}</h1>
<h1>{{ message.split('').reverse().join('') }}</h1>
```

### Event Listeners in Vue
- **`v-on` 디렉티브**(축약형 `@`)로 DOM 이벤트를 수신
- 이벤트 핸들러에서 상태를 변경하면 화면이 자동 업데이트

```html
<button v-on:click="increment">{{ count }}</button>
```
```js
setup() {
  const count = ref(0)
  const increment = function () {
    count.value++
  }
  return { count, increment }
}
```

### [참고] ref는 왜 객체인가?
- 일반 변수가 아닌 **객체 데이터 타입**으로 만든 이유:
  - JS에서 **일반 변수의 접근/변경은 감지할 방법이 없음**
  - 반면 **객체의 속성 접근(get)과 변경(set)은 가로챌 수 있음**
- 그래서 `.value` 속성의 **get에서 추적(track)**, **set에서 변경 알림(trigger)** 을 수행 → 값이 바뀌면 자동으로 DOM 업데이트

## 4. Single-File Components

### Component
- **재사용 가능한 코드 블록**
- UI를 독립적이고 재사용 가능한 조각으로 분할 → 앱은 **중첩된 컴포넌트 트리** 구조가 됨
  - 예: `<Root>` → `<Header>` / `<Main>` / `<Aside>` → `<Article>×2`, `<Item>×3`
- 실제 서비스 예: 음악 앱에서 같은 형태의 카드 UI가 데이터만 바뀌며 반복되는 것

### SFC (Single-File Components)
- 컴포넌트의 **템플릿(HTML), 로직(JS), 스타일(CSS)을 하나의 `*.vue` 파일에 캡슐화**한 특수 파일 형식

### SFC 문법 — 3가지 최상위 언어 블록

```vue
<template>
  <div class="greeting">{{ msg }}</div>
</template>

<script setup>
import { ref } from 'vue'
const msg = ref('Hello World!')
</script>

<style scoped>
.greeting {
  color: red;
}
</style>
```

| 블록 | 설명 |
|---|---|
| `<template>` | 각 `*.vue` 파일은 최상위 template 블록을 **하나만** 포함 가능 |
| `<script setup>` | 각 `*.vue` 파일은 **하나의** `<script setup>` 블록만 포함 가능. 컴포넌트의 `setup()` 함수로 사용되며, 컴포넌트 인스턴스가 생성될 때마다 실행 |
| `<style scoped>` | `scoped`가 지정되면 CSS가 **현재 컴포넌트에만 적용**됨 |

### 컴포넌트 사용해보기
- https://play.vuejs.org 에서 Vue 컴포넌트 코드 작성·미리보기 가능
- 실제 프로젝트에서는 **Vite** 같은 공식 빌드 도구를 사용

## 5. Node.js

### Node.js
- **Chrome의 V8 JavaScript 엔진 기반의 Server-side 실행 환경(런타임)**
- Node.js의 영향
  1. 기존에 브라우저 안에서만 동작하던 JS를 **서버 측에서도 실행 가능** → 프론트엔드와 백엔드에서 **동일 언어 사용**
  2. **NPM을 활용해 수많은 오픈소스 패키지·라이브러리 공유** → 개발자들이 손쉽게 코드를 공유·재사용

- **Node.js의 기본 패키지 관리자**

## 6. SFC build tool — Vite

### Module (모듈)
- **필요성**: 앱이 커지고 복잡해지며 하나의 파일에 모든 기능을 담기 어려움 → 파일을 여러 개로 분리해 관리 → **분리된 각 파일이 모듈** (기능별로 분리된 독립적인 파일)
- **한계**: 앱이 발전하며 처리할 JS 모듈 수가 극적으로 증가 → 성능 병목, **모듈 간 의존성(연결 관계)** 이 깊어져 파악하기 어려움 → 해결책: **Bundler**
  - (밈: "우주에서 가장 무거운 물체 = node_modules")

### Bundler (번들러)
- **여러 모듈과 파일을 하나(혹은 여러 개)의 번들로 묶어 최적화**하여 앱에서 사용할 수 있게 만들어주는 도구
- 역할: 의존성 관리, 코드 압축·최적화, 리소스 처리 등
- [참고] **Vite는 Rollup이라는 Bundler를 사용**하며, 개발자가 환경 설정에 신경 쓰지 않도록 미리 설정해 둠

### Vite
- 프론트엔드 개발 도구 — https://vitejs.dev
- **빌드(build)**: 파일·리소스(JavaScript, CSS, 이미지 등)를 **배포 가능한 소프트웨어 제품으로 변환·생성**하는 과정

### Vite 튜토리얼 (Vue 프로젝트 생성)
```bash
npm create vue@latest   # 1. 프로젝트 생성 (이름 입력, 기능 선택 — 기본은 모두 No)
cd vue-project          # 2. 프로젝트 폴더 이동
npm install             # 3. 의존성 패키지 설치
npm run dev             # 4. 개발 서버 실행 → http://localhost:5173
```
- 실행 후 "You did it! — Vite + Vue 3" 시작 페이지 확인

## 7. Vue 프로젝트 구조

```text
VUE-PROJECT
├── node_modules/
├── public/
├── src/
│   ├── assets/
│   ├── components/
│   ├── App.vue
│   └── main.js
├── .gitignore
├── index.html
├── jsconfig.json
├── package-lock.json
├── package.json
├── README.md
└── vite.config.js
```

| 항목 | 역할 |
|---|---|
| `node_modules/` | 프로젝트에서 사용하는 **외부 패키지들이 저장**되는 디렉토리. `npm install`로 언제든 재생성 가능 → **`.gitignore` 대상** |
| `package.json` | 프로젝트의 **메타 정보(이름, 버전, 작성자 등)와 의존성 패키지 목록** |
| `package-lock.json` | 패키지들의 **실제 설치 버전·하위 의존성을 정확히 고정** → 여러 개발자가 **일관성 있는 의존성**으로 설치 가능. `npm install` 시 이 파일 기준으로 설치 |
| `public/` | 소스코드에서 참조하지 않는 **정적 파일** 보관. 빌드 시 처리 없이 그대로 복사됨 |
| `src/` | 프로젝트의 **소스코드** |
| `src/assets/` | 프로젝트에서 사용하는 자원(이미지, 폰트, 스타일시트 등). **컴포넌트에서 참조하는 내부 파일** 저장, 빌드 시 처리됨 |
| `src/components/` | **Vue 컴포넌트들을 작성**하는 곳 |
| `src/App.vue` | 앱의 **Root(최상위) 컴포넌트**. `<style scoped>`를 쓰면 App 컴포넌트 내부에만 스타일이 적용되고, 전역 레이아웃 스타일은 `<style>`로 남길 수 있음 |
| `src/main.js` | 앱의 **JS 진입점** — 앱 인스턴스를 생성하고 mount 실행 |
| `index.html` | Vue 앱의 **기본 HTML 파일이자 진입점(entry point)**. Root 컴포넌트 `App.vue`가 이 페이지에 **mount**됨 → **Vue 앱이 SPA인 이유**. bootstrap 같은 외부 리소스(CSS·JS)를 로드할 수 있음 |
| `jsconfig.json` | 컴파일 옵션, 모듈 경로 별칭(`@` = `src/`) 등 설정 |
| `vite.config.js` | **Vite 설정 파일** — 플러그인, 빌드 옵션 등 |

## 8. Vue Component 활용

### 컴포넌트 사용 2단계
**① 정의(생성)** — `src/components/MyComponent.vue` 생성
```vue
<template>
  <div>
    <h2>MyComponent</h2>
  </div>
</template>

<script setup>
</script>
```

**② 등록(import) 후 사용** — `App.vue`
```vue
<template>
  <h1>App.vue</h1>
  <MyComponent />
</template>

<script setup>
import MyComponent from '@/components/MyComponent.vue'
</script>
```
- `@`는 `src/` 경로 별칭
- Vue devtools의 Components 탭에서 `<App>` → `<MyComponent>` 트리 구조 확인 가능

### 컴포넌트 중첩과 재사용
- 컴포넌트 안에서 또 다른 컴포넌트를 import해 사용 가능
- 같은 컴포넌트를 **여러 번 재사용** 가능

```vue
<!-- MyComponent.vue -->
<template>
  <h2>MyComponent</h2>
  <MyComponentItem />
  <MyComponentItem />
  <MyComponentItem />
</template>

<script setup>
import MyComponentItem from '@/components/MyComponentItem.vue'
</script>
```

### [참고] 컴포넌트 이름 지정
- 스타일 가이드(Strongly Recommended)에 따라 **PascalCase** 등 이름 규칙 준수 권장
- https://vuejs.org/style-guide/rules-strongly-recommended.html

## 9. 추가 주제 & 참고

### Composition API vs Option API
Vue 컴포넌트를 작성하는 2가지 스타일

| | Composition API | Option API |
|---|---|---|
| 작성 방식 | `import`한 API 함수들(`ref`, `onMounted` 등)을 사용해 로직 정의 | `data`, `methods`, `mounted` 같은 **옵션 객체**로 로직 정의 |
| 주 사용처 | `<script setup>`과 함께 사용 | 옵션에 정의된 속성은 `this`로 접근 |

```vue
<!-- Composition API -->
<script setup>
import { ref, onMounted } from 'vue'

const count = ref(0)
function increment() {
  count.value++
}
onMounted(() => {
  console.log(`Count is: ${count.value}`)
})
</script>

<template>
  <button @click="increment">{{ count }}</button>
</template>
```

```vue
<!-- Option API -->
<script>
export default {
  data() {
    return { count: 0 }
  },
  methods: {
    increment() {
      this.count++
    }
  },
  mounted() {
    console.log(`Count is: ${this.count}`)
  }
}
</script>

<template>
  <button @click="increment">{{ count }}</button>
</template>
```

**권장 사항**
- **Composition API + SFC**: 규모가 있는 앱의 전체를 빌드하려는 경우
- **Option API**: 빌드 도구를 사용하지 않거나 복잡성이 낮은 프로젝트

### SEO (Search Engine Optimization)
- Google, Naver 같은 **검색 엔진에 내 서비스·제품이 더 잘 노출되도록 개선하는 과정**
- 검색 엔진이 분석하는 정보의 대상은 주로 **HTML 내용**
- 최근 SPA·CSR 서비스 비중이 증가 → SPA도 검색 대상이 되도록 **SEO를 지원하는 방향으로 발전 중**

### CSR & SSR — 무엇이 정답?
- 정답은 없음 — **앱의 목적, 규모, 성능, SEO 요구사항**에 따라 달라짐
- SPA에서도 SSR을 지원하는 프레임워크 존재
  - Vue → **Nuxt.js** / React → **Next.js**

### Single Root Element
- 가독성, 스타일링, 명확한 컴포넌트 구조를 위해 각 컴포넌트에는 **단일 루트 요소(Single Root Element)** 사용을 권장
- 템플릿 최상위에 여러 요소를 나열하기보다 하나의 요소(예: `<div>`)로 감싸는 것

### Scoped Style
- `<style>`에 `scoped`를 붙이지 않으면 스타일이 **전역으로 적용** → 다른 컴포넌트의 같은 태그(예: `div`)에도 스타일이 함께 적용됨
- **부모-자식 관계에서의 스타일 전파**
  - 일반적으로 scoped 스타일은 자식 컴포넌트에 영향을 미치지 않음
  - **예외**: 자식 컴포넌트의 **최상위 요소(root element)** 는 부모와 자식 CSS **모두의 영향을 받음**
  - 부모가 레이아웃 목적으로 자식의 루트 요소를 스타일링할 수 있도록 **의도적으로 설계**된 동작
- 권장: 상위 App 컴포넌트의 레이아웃 스타일은 전역일 수 있지만, **다른 모든 컴포넌트는 scoped 스타일 사용** (스타일 가이드 Essential 규칙)

### 패키지 관리 주의사항
- **`node_modules` 직접 수정·커밋 금지** — `npm install`로 언제든 재생성 가능하므로 `.gitignore`에 포함
- **`package.json` / `package-lock.json` 직접 편집 자제** — `npm install 패키지명` 등 명령을 통해 자동 업데이트되도록
- **문제 발생 시 재설치** — 패키지 버전 충돌이나 이상 동작이 의심되면 `node_modules` 삭제 후 `npm install` 재실행
